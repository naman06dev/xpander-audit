Summary

The biggest thing that came through in our audit is cohesion — or the lack of it. There's genuinely good content scattered across these 153 pages (the Self-Hosted deployment guide is excellent, the Workflows explainer does a great job of motivating the "why," the Build section has real UI walkthroughs with screenshots), but none of it connects into a single experience that someone can follow from start to finish. A developer, a PM using the Workbench, and a platform engineer all land on the same 26-card grid and have to figure out on their own what's relevant to them.

And then when a developer does find the getting started examples and tries to run them, the code crashes on the current SDK version. The terminology for the same concept changes between pages ("Cultural Knowledge" on one page becomes "Agent Memories" on another, and good luck searching for either). So even the content that exists in isolation doesn't hold together when someone tries to move through the docs as a whole.

The product underneath is solid (we verified: the CLI scaffolds working code, the REST API responds in under 6 seconds, multi-framework support across 7 frameworks actually works). The new landing page is already moving toward the right framing with three products: AI Workflows, Personal AI, and AI Agents. The docs just need to catch up to where the product already is, and this report lays out how we'd do that.

Code Verification

Fresh environment. SDK v2.0.249, CLI v0.0.69. Ran Examples 00, 01, and 05 verbatim.

Python 3.12.0 | Node.js v22.20.0 | macOS (Darwin 23.5.0)
pip install "xpander-sdk[agno]"     # v2.0.249
npm install -g xpander-cli          # v0.0.69
xpander login --api-key "..."       # Authenticated successfully

Installation and authentication: no issues.

The Hello World crashes Fatal 
# Exact code from the docs:
backend = Backend()
agno_agent = Agent(**backend.get_args())  # crashes here

AttributeError: 'NoneType' object has no attribute 'instructions_override'
xpander_sdk/.../agno.py line 140 — task is None outside @on_task handler

The SDK accesses task.instructions_override without checking if task exists. When you call get_args() outside of an @on_task handler — exactly how the tutorial uses it — task is None and the SDK crashes. Example 05 (Custom Tools) has the same crash.

The working pattern already exists

xpander agent init scaffolds a handler using Backend(configuration=task.configuration) and aget_args(task=task), which works. Fix: update the SDK to handle task=None, or update the tutorials to show the pattern the CLI already generates.

xpander init doesn't exist High

Tutorials say xpander init. Correct command is xpander agent init [agent-id]. The CLI falls through to the help menu (which does list the correct command), but no initialization happens and there's no error.

No explanation of where to get credentials High

Every example loads XPANDER_API_KEY, XPANDER_ORGANIZATION_ID, and XPANDER_AGENT_ID from .env. No page explains where to find these. xpander agent init auto-generates a filled .env, but the docs point to the wrong command.

The 26-Card Problem

This is what the User Guide looks like today. Everything the platform can do, in one flat grid:

The Build section alone has 12 cards. The full User Guide has 26. No routing, no paths, no "if you're X, start here."

This is the surface area of the product. It's impressive — there's a lot here. But four types of people all land on this same grid and have to figure out what's relevant to them.

Developers Partially served
Python SDK, CLI, custom tools. Content exists but examples are broken and credentials aren't explained.
Builders Underserved
PMs, data scientists, ops. The Build section has screenshots and walkthroughs — but no path routes them there from the homepage.
Platform Engineers Barely served
K8s deployment. The Self-Hosted page is great, but no scaling guides, monitoring, or upgrade paths.
AI Leadership Not served
Architecture overviews, security/compliance. SOC 2 Type II on the marketing site, nothing in the docs.

Solution: Fork early, bridge often

Two-path fork on the landing page: Workbench (visual) and SDK (code). Platform engineers enter through the code path and branch to Self-Hosted. Leadership gets an architecture page both paths link to.

Terminology

The same concept has different names on different pages. This makes it hard to search, hard to connect ideas, and hard to build a mental model.

Concept Names used across docs
Agent integrations Tools, Connectors, Tools & Connectors, Connectors (Tools), Toolkits
Shared agent memory Cultural Knowledge, Agent Memories, Agentic Context
The product xpander.ai, Xpander, xpander, XPANDER
API endpoint Unified API, REST API, Platform REST API

Fix: Create a glossary with one canonical name per concept. Find-and-replace across all 153 pages. This is mechanical — no new content needed, just consistency. Every other recommendation in this document works better once the vocabulary is stable.

What's Working

  • Self-Hosted Deployment is the best page in the docs. Full Helm commands, YAML values, kubectl verification, health checks, troubleshooting.
  • Workflows page nails the "why." Its "Why Agentic Design?" section is the standard for concept pages.
  • Build section has real UI content. The Agent page walks through the Workbench wizard with 4+ screenshots. Connectors page shows auth flows visually. There's a "Visual Agent Builder to Code" 5-min win.
  • Examples 09 and 12 show what tutorials should look like. Real architecture, confidence thresholds, expected behavior.
  • Multi-framework support is a real differentiator. Same agent across 7 frameworks.
  • REST API reference is solid. 33 endpoints, 3-language examples, rate limits.
  • Changelog actively maintained (latest: Feb 22, 2026).
  • llms.txt already exists. Ahead of most competitors on AI discoverability.

Other Issues

Content Gaps

  • No architecture page. No diagram of how agents, tasks, tools, and frameworks connect.
  • No troubleshooting section. Only the Self-Hosted page has one.
  • No security/compliance page. SOC 2 Type II is on the marketing site but not in docs.
  • No migration guide. Moving between frameworks has no path.

Developer Experience

  • CLI requires both Node.js 20+ and Python 3.12+. Unusual dual-runtime, never explained.
  • CLI Reference lists 3 frameworks. User Guide lists 5.
  • No example shows expected terminal output.
  • Two different Backend() patterns (env-var vs explicit config), never explained when to use which.

Writing & Navigation

  • Platform Overview tagline: "full-stack, multi-framework, multi-cloud, self-hosted agent platform." Five qualifiers in a row.
  • No persistent sidebar — only card grids and breadcrumbs.
  • 5-Minute Wins buried under Examples instead of being a front-door entry point.
  • Examples numbered 00, 01, 04, 05, 06... The gaps at 02/03 look unfinished.
